## Two repos, two pushes

This blog lives in its own repository, and my homelab consumes it as a pinned flake input.
That pin is exact --- the homelab deploys the specific commit recorded in its `flake.lock`,
not whatever happens to be on master.

Which meant publishing a post used to take two trips. Push the post to the blog repo. Then
walk over to the <GitHubRepo repo="smg1024/homelab" /> repo, run `nix flake update blog`,
commit, and push again. Two repos, two pushes, for one blog post.

I wasn't happy with that flow. And I had a second motive: I wanted to practice CI/CD
plumbing across repositories --- the kind of cross-repo wiring that real infrastructure work
is full of.

## Cron or dispatch

I considered two designs. One was a nightly cronjob at 03:00 KST in the homelab repo:
check the blog's master branch for new commits, re-pin if it moved, every day. The other
was event-driven: the blog repo tells the homelab the moment master changes.

<Decision title="Event-driven dispatch">
  Latency and learning value tipped it. A post should be live minutes after I push, not the next
  morning --- and building the dispatch mechanism was the more interesting thing to learn.
</Decision>

## The trigger

The blog's side is one small workflow: every push to master sends a `repository_dispatch`
event to the homelab repo.

```yaml title=".github/workflows/trigger-homelab-deploy.yml"
name: Trigger homelab deploy

on:
  push:
    branches: [master]

jobs:
  dispatch:
    runs-on: ubuntu-latest
    steps:
      - name: Send repository_dispatch to smg1024/homelab
        env:
          GH_TOKEN: ${{ secrets.HOMELAB_PAT }}
        run: gh api repos/smg1024/homelab/dispatches -f event_type=blog-updated
```

<Note title="Github PAT (Personal Access Token)">
  Cross-repo dispatch needs a credential. I minted a fine-grained personal access token scoped to
  only the homelab repository, with only contents and pull-request permissions, and stored it as an
  Actions secret in the blog repo. I didn't set an expiration date --- I'm relying on GitHub's
  security around secret management here.
</Note>

## The receiving end

When the `blog-updated` event lands, the homelab's `update-blog` workflow takes over. It
runs `nix flake update blog` and checks the git diff first --- if nothing changed, the
workflow ends right away. If `flake.lock` did move, it commits the change to a separate
branch on my behalf, opens a PR, and auto-merges it to main once all CI checks pass. From
there it's the same deploy action every other homelab change goes through.

## Guardrails for an audience of one

Even the bot's one-line `flake.lock` bump goes through a branch, a PR, and CI before it
touches main. Branch protection rules enforce that --- no exceptions, not even for my own
automation. The gate is there in case the blog fails to deploy, especially when a
dependency changes in the blog project: CI catches the broken build before any node
rebuilds.

This might look somewhat overcomplicated for a simple blog project. But if I assume this
was a non-personal, team project, this is the standard move.
